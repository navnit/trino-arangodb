# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A Trino connector plugin (`packaging: trino-plugin`) that lets Trino run SQL queries against ArangoDB. ArangoDB databases map to Trino schemas, collections map to tables. Currently at milestone M4 ("structured-type materialization"): read-only, with equality/range filter and limit pushdown (M1/M2), plus per-shard parallel splits on cluster deployments for eligible (non-smart, multi-shard hash) collections (M3); recursive ARRAY/ROW/DECIMAL value materialization (M4).

## Commands

Maven is not on `PATH` by default in this environment; if `mvn` reports "command not found", run:

```bash
source ~/.sdkman/bin/sdkman-init.sh
```

Build requires Java 24 (`maven.compiler.release=24` in `pom.xml`).

```bash
mvn package                          # build the trino-plugin artifact
mvn test                             # run the full test suite
mvn test -Dtest=TypeMapperTest       # run a single test class
mvn test -Dtest=TypeMapperTest#mergeIntAndFloatWidensToDouble   # single test method
```

Most test classes (`ArangoClientTest`, `ArangoConnectorQueryTest`, `ArangoPageSourceProviderTest`, and others using `TestingArangoServer`) spin up a real ArangoDB container via Testcontainers — **Docker must be running locally** for `mvn test` to pass. No mocking framework is used anywhere in the test suite; where a test does need to avoid hitting a live server (e.g. `ArangoMetadataTest`'s error-path cases), it uses hand-written `ArangoClient` subclasses as test doubles rather than a container. End-to-end SQL queries run against a live container via Trino's `DistributedQueryRunner` (see `ArangoConnectorQueryTest`).

There is no linter/formatter plugin configured in `pom.xml`.

### pom.xml is a standalone module with no parent POM

This means several dependency versions that a real Trino connector would normally inherit from `io.trino:trino-root`'s `dependencyManagement` are pinned explicitly here instead, each with a long comment explaining *why* (`trino-spi`-transitive `provided`-scope pins for `jackson-annotations`/`opentelemetry-api`/`opentelemetry-context`; `io.airlift` `jaxrs`/`http-client`/`jmx` pinned to `336` to fix a `discovery-server:1.37`-induced skew; `junit-jupiter-api`/`junit-platform-launcher` pins to fix JUnit5 mediation). **Do not remove or "simplify" these pins** without reading the comment above each one — they exist to work around real dependency-mediation bugs, not stylistic preference. The Surefire `argLine` and `api.version=1.51` system property are similarly required workarounds (the latter for Testcontainers 1.20.4's shaded docker-java client rejecting modern Docker Engine's default API version) — also documented inline.

## Architecture

### SPI wiring (Plugin → Connector)

Trino discovers the connector via `ArangoPlugin.getConnectorFactories()` → `ArangoConnectorFactory` (registered under the name `"arangodb"`). `ArangoConnectorFactory.create()` boots an Airlift `Bootstrap`/Guice `Injector` from `ArangoModule`, then pulls a singleton `ArangoConnector` out of it. `ArangoModule` binds `ArangoClient`, `TypeMapper`, `SchemaResolver`, `AqlBuilder`, `ArangoMetadata`, `ArangoSplitManager`, `ArangoPageSourceProvider`, and `ArangoConnector` as `Scopes.SINGLETON`; `ArangoConfig` is bound separately via Airlift's `configBinder(...).bindConfig(...)`. `ArangoConnector` just hands out these singletons for `getMetadata()`/`getSplitManager()`/`getPageSourceProvider()`, and its `shutdown()` calls `LifeCycleManager.stop()`, which triggers `ArangoClient`'s `@PreDestroy` (`close()` on the underlying `ArangoDB` driver instance).

`ArangoConfig` (bound via Airlift's `@Config`) holds every user-facing setting: `arangodb.hosts` (comma-separated `host:port` list), `arangodb.user`, `arangodb.password` (`@ConfigSecuritySensitive`), `arangodb.schema.sample-size` (default 1000), `arangodb.schema.sample-random`, `arangodb.schema.mixed-type-strategy` (`VARCHAR` or `JSON`, though `JSON` is not actually wired up yet — see `TypeMapper.merge`), `arangodb.type-coercion` (`lenient`, the default, or `strict` — see `ArangoPageSource.appendValue` and `ArangoMetadata.applyFilter` below), `arangodb.shards-per-split` (default `1` — target number of shards grouped into each split on cluster fan-out), `arangodb.max-splits` (default `32` — hard cap on the number of splits per collection scan; this cap can force more shards into a single group than `arangodb.shards-per-split` once `ceil(shardCount / shardsPerSplit)` exceeds it), and `arangodb.shard-parallelism-enabled` (default `true` — set to `false` to force single-split scans unconditionally and never invoke the internal `shardIds` option; see `ArangoSplitManager` below).

### Read path (Metadata → SplitManager → PageSourceProvider → PageSource)

This is the part that spans the most files and is worth understanding as one flow:

1. **`ArangoMetadata`** (`ConnectorMetadata`) — `listSchemaNames` = ArangoDB databases, `listTables`/`getTableHandle` = non-system collections within a database. Column info comes from `resolve()`, which calls `SchemaResolver.resolveColumns()` and memoizes the result per `SchemaTableName` in an unbounded `ConcurrentHashMap` (a real TTL cache is deferred past M1).
2. **`SchemaResolver`** samples up to `sampleSize` documents from the collection (`ArangoClient.sampleDocuments`) and infers a schema by taking the *union* of fields across all sampled docs, merging types per field via `TypeMapper.merge`. System attributes (`_key`, `_id`, `_rev`) are always added as hidden `VARCHAR` columns; `_from`/`_to` are added as visible `VARCHAR` columns only when the collection is an edge collection. A field seen only as `null` across the whole sample resolves through the `UnknownType.UNKNOWN` bottom sentinel down to `VARCHAR` (recursively, even nested inside `ROW`/`ARRAY`) — see `SchemaResolver.resolveUnknown`.
3. **`TypeMapper.inferType`/`merge`** implement the type-inference rules: `Boolean→BOOLEAN`, `Integer/Long→BIGINT`, `BigInteger` beyond 63 bits or `uint64→DECIMAL(38,0)`, other `Number→DOUBLE`, `String→VARCHAR`, `List→ARRAY(...)` (recursively merging element types), `Map→ROW(...)`. Merging an int-typed and double-typed occurrence of the same field widens to `DOUBLE`; merging any other genuinely incompatible pair falls back to `VARCHAR` regardless of `mixed-type-strategy` (the `JSON` strategy value exists in config but isn't implemented as an actual output type yet).
4. **`ArangoSplitManager`** decides, per table on every `getSplits` call, whether to fan a collection scan out into multiple `ArangoSplit`s (one per shard group) or fall back to a single split that scans the whole collection. When `arangodb.shard-parallelism-enabled` is `false` it always returns a single split. Otherwise it runs a pipeline implemented in the new `io.arango.trino.split` package: **discover** the collection's `ShardingInfo` (shard count, sharding strategy, SmartJoin attribute) → **allowlist gate** (`ShardEligibility`) — only non-smart hash-sharded collections (`hash`/`community-compat`/`enterprise-compat`) with more than one shard and no `smartJoinAttribute` are eligible; SmartGraph/SmartJoin collections are always excluded because their edges can live in multiple internal sub-shards, which would double-count rows under naive per-shard scanning → **enumerate** the collection's shard IDs → **group** them (`ShardGrouping.partition`) into `min(ceil(shardCount / arangodb.shards-per-split), arangodb.max-splits)` balanced groups, with every shard landing in exactly one group → **probe** (`ShardFanoutCapability`) whether the server can be trusted to honor the internal `shardIds` query option — a version pin (server must report ≥ 3.11) plus an active count-sum probe (Σ of per-group `shardIds`-scoped counts must equal the full collection count) for the actual groups about to be emitted, computed once per connector process and cached (an inconclusive probe, e.g. an empty collection, is retried rather than cached) → **emit** one `ArangoSplit` per shard group. A failure anywhere in this pipeline (ineligible collection, unconfirmed capability, or a discovery exception) falls back to a single split scanning the whole collection; a multi-shard collection that falls back this way logs a `WARN` so the fallback is observable, while a genuinely non-sharded/single-shard collection falls back silently at `DEBUG`.
5. **`ArangoMetadata.applyFilter`** pushes a `Constraint` down into the `ArangoTableHandle`'s stored `TupleDomain`, per-column, via `isPushable`: equality/IN (`ValueSet.isDiscreteSet()`) is pushable for `BOOLEAN`/`VARCHAR`/`BIGINT`/`DOUBLE`; numeric range is pushable only for `BIGINT`/`DOUBLE`. `AqlBuilder` renders each pushed comparison so the AQL predicate can't be satisfied by a value the read path would reject or read differently: `BOOLEAN`/`VARCHAR`/`BIGINT` compare bare (AQL `==`/`IN`/`<`/`>` are type-strict and the `BIGINT` read path is exact — `longValue()`, no rounding); `BIGINT` range guards only with a bare `IS_NUMBER(...)` (no integrality conjunct — AQL `FLOOR()` returns a `double`, so `d.f == FLOOR(d.f)` would false-miss a stored `int64` beyond 2⁵³ that reads back exactly, review finding C3; the guard doesn't need to exclude fractionals because `BIGINT` range is *prefilter-only* — see below — and the residual re-check drops them); and every `DOUBLE` comparison (eq/IN **and** range) is rendered as `IS_NUMBER(d.f) AND (d.f + 0.0) <op> @v` — the `+ 0.0` promotes a stored `int64` into double space so AQL compares exactly what `appendValue` reads via `doubleValue()` (ArangoDB otherwise compares `int64`-vs-`double` by exact mathematical value, which diverges from the rounded read path beyond 2⁵³ — review finding C1), and the `IS_NUMBER` guard precedes the arithmetic because `+ 0.0` coerces non-numbers. The promotion is `DOUBLE`-only: promoting `BIGINT` would round its exact read path and reintroduce the same class of bug in reverse. **Known limitation:** the `+ 0.0` arithmetic also nullifies a stored *non-finite* `double` (`Infinity`/`NaN`), so a fully-pushed `DOUBLE` predicate omits such a row server-side even though the read path would keep it. This is unreachable through normal JSON ingestion — ArangoDB cannot represent `Infinity`/`NaN` in JSON (it renders them as `(non-representable type double)`) — and can only occur for documents written by a native-VelocyPack driver; it is an accepted limitation (surfaced by the C3 re-review), not closed in M2. `IS NULL` (a null-allowing domain, caught by `isPushable`'s `domain.isNullAllowed()` check) always stays residual — AQL's null test operates on the raw stored value, which can diverge from Trino's post-coercion null-ness on a type mismatch. `IS NOT NULL` also always stays residual, but via the separate `values.isAll()` check just after it (its domain excludes null, so it never trips `isNullAllowed()`); `VARCHAR` range stays residual too, falling through to `isPushable`'s final type check since only `BIGINT`/`DOUBLE` range is pushable. Because the bare `IS_NUMBER` guard admits fractional values and integral values outside signed-64-bit range (both of which `appendValue` reads as `NULL`), a `BIGINT` range push is *prefilter-only*: it's applied in AQL **and** kept in Trino's residual for a post-read re-check (the constraint is not fully pushed down), whereas equality/IN and `DOUBLE` range are fully enforced by the pushed AQL alone (fully pushed down, nothing left residual). When `arangodb.type-coercion=strict`, `isPushable` always returns `false` — pushing a filter would let ArangoDB silently exclude a type-mismatched row server-side, which would suppress the `ARANGODB_TYPE_CONVERSION_ERROR` strict mode is supposed to raise, so strict mode declines all pushdown and evaluates every constraint as residual instead. `ArangoMetadata.applyLimit` similarly pushes a `LIMIT` into the handle, but `limitGuaranteed` now tracks `!config.isShardParallelismEnabled()`: with shard-parallelism enabled (the default), `ArangoSplitManager` may fan a collection out into multiple splits, each executing its own `LIMIT n` independently, so a pushed limit is a per-split reduction rather than a global guarantee and Trino must still enforce the final `LIMIT` itself over the merged results; only with `arangodb.shard-parallelism-enabled=false` (always single-split) is the pushed limit exact.
6. **`ArangoPageSourceProvider.createPageSource`** builds the AQL query via `AqlBuilder.buildScan`, which starts from a base `FOR d IN @@col` scan and appends a `FILTER` clause when the handle has a pushed `TupleDomain` (predicate values are bind-parameterized through `bindVars`) and a `LIMIT` clause when the handle has a pushed limit (the limit is appended as a literal integer directly into the AQL string, *not* bind-parameterized). The query always ends in a `RETURN {"col": d["col"], ...}` clause built by `buildReturnClause`, projecting exactly the requested columns as an object literal — there is no bare `RETURN d` form, even for an unfiltered/unlimited full scan. It constructs an `ArangoPageSource` over the resulting cursor. Structured columns (`ARRAY`/`ROW`/`DECIMAL`) materialize like any other since M4 — there is no up-front rejection anymore.
7. **`ArangoPageSource`** streams `ArangoCursor<Map>` into Trino `Page`s in batches of 1024 rows. Value coercion (`appendValue`) is *type-exact*: a stored value whose runtime type doesn't match the column's inferred Trino type is a mismatch, not something to coerce — no more `String.valueOf` or `longValue()` truncation. Under `arangodb.type-coercion=lenient` (the default) a mismatch reads as `NULL`; under `strict` it raises `TrinoException(ARANGODB_TYPE_CONVERSION_ERROR, ...)`. This is a deliberate M1 read-behavior change: a number stored under a `VARCHAR` column, or a fractional value (`42.5`) stored under a `BIGINT` column, now reads as `NULL` instead of being coerced. A `BIGINT` column does still accept a fractional-*free* `Double`/`Float`/`BigInteger` (e.g. stored `42.0` reads as `42`) — only genuine fractions are a mismatch; truncating them would silently disagree with a pushed range filter that already excluded them, which is the exact class of bug type-exactness closes. This exactness is also what lets `applyFilter` push equality/range predicates safely at all: the pushed AQL comparison and this read path are guaranteed to admit exactly the same values. Coercion of structured values lives in `io.arango.trino.type.ValueMaterializer` (`TypeMapper`'s read-side dual; `ArangoPageSource` itself stays pure cursor/paging and delegates to it) — materialization is recursive and leaf-level exact: a nested mismatch nulls only the offending element/field under `lenient` and raises `ARANGODB_TYPE_CONVERSION_ERROR` with a path to the offending leaf (e.g. `col[2].b`) under `strict`; an absent `ROW` field or a stored `null` at any depth is `NULL` in both modes, never treated as a mismatch; `DECIMAL(38,0)` accepts any integral number gated by `Decimals.overflows(..., 38)` — deliberately *not* the ±2⁶³ `BIGINT` bound, since uint64-range values like `1e19` must still read back correctly.

### Error translation

`ArangoMetadata` distinguishes "the ArangoDB database doesn't exist" (driver error number 1228, checked via `isDatabaseNotFound`) from every other `ArangoDBException`. The former is treated as Trino's normal "schema/table not found" (return `null` / skip), while everything else (auth failure, network partition, etc.) is rethrown as `TrinoException(GENERIC_INTERNAL_ERROR, ...)` so real failures aren't silently swallowed as "not found."

### `UnknownType` relocation

`io.arango.trino.type.UnknownType` is a verbatim Apache-2.0 relocation of Trino's own `io.trino.type.UnknownType`. It had to be copied rather than depended on: Trino's real `UnknownType` lives in the `trino-main` query-engine module, not in the public `trino-spi` contract a connector plugin can depend on, and Trino's plugin classloader only exposes `trino-spi` classes to plugins — a direct reference would fail with `NoClassDefFoundError` at runtime even if it compiled. This is documented in the class's own Javadoc and in `NOTICE`.

### Package layout

- `io.arango.trino` — SPI implementation classes (`ArangoPlugin`, `ArangoConnectorFactory`, `ArangoModule`, `ArangoConnector`, `ArangoConfig`, `ArangoMetadata`, `ArangoSplitManager`, `ArangoPageSourceProvider`, `ArangoPageSource`, `ArangoTransactionHandle`)
- `io.arango.trino.client` — `ArangoClient`, the sole wrapper around the ArangoDB Java driver (`ArangoDB`/`ArangoCursor`); also holds test-only seeding helpers (`createDatabaseForTest`, `insertForTest`, etc.) used by both unit and integration tests
- `io.arango.trino.handle` — SPI handle records (`ArangoTableHandle`, `ArangoColumnHandle`, `ArangoSplit`), all Jackson-serializable
- `io.arango.trino.schema` — `SchemaResolver`, the sampling/inference pipeline
- `io.arango.trino.type` — `TypeMapper`, `ValueMaterializer`, and the relocated `UnknownType`
- `io.arango.trino.aql` — `AqlBuilder`, translates a table scan into an AQL query string + bind vars
- `io.arango.trino.split` — shard-parallel fan-out logic: `ShardingInfo` (record: shard count / sharding strategy / SmartJoin attribute), `ShardEligibility` (the non-smart-hash allowlist gate), `ShardGrouping` (partitions shard IDs into balanced split groups), `ShardFanoutCapability` (version pin + lazy-cached active count-sum probe gating use of the internal `shardIds` option)

`TestingArangoServer` (`src/test`) wraps the Testcontainers-managed ArangoDB instance shared by integration-style tests; `TestingArangoCluster` (`src/test`) similarly wraps a real multi-node ArangoDB cluster (coordinator/agency/dbservers, via Testcontainers' Compose support) used by the shard-parallel cluster ITs (`ShardParallelClusterIT`, `ShardParallelCorrectnessIT`).
